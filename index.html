Guia rápido para Git/Github

Link do curso.

  • Iniciar um repositório: 
    $ git init
  • Verificar o staus do repositório: 
    $git status
  • Adicionar arquivos ao controle de versão: 
    $ git add "nome_do_arquivo"
    ou para todos: 
    $ git add *
    ou 
    $ git add .
  • Criar versões do código com commit 
    $ git commit -m "commit inicial"
    Caso seja necessário, o Git vai pedir pra vc configurar seus dados:
    • Configurar e-mail: 
      $ git commit --global user.email "seuemail@provedor.com"
    • Configurar seu Usernam: 
      $ git config --global user.name "seuUserName"
  • Enviar as alterações registradas no commit. No Github, o repositório "em nuvem" já deve ter sido criado. Para esse guia, o repositório "https://github.com/renzoalves/aprendendo-git"
    1. Adicionando o repositório remoto (sem as aspas): 
      $ git remote add origin "https://github.com/renzoalves/aprendendo-git"
    2. Definir a branch para qual deverá ser enviado os arquivos. Onde "master" é o nome da branch. 
      $ git push --set-upstream origin master
  • Ciclo de atualização de código
    • Adicione novamente os arquivos: 
      $ git add .
    • Verifique se os arquivos que foram atualizados constam na lista: 
      $ git status
    • Agora commite novamente: 
      $ git commit -m "Atualização 2022.03.29"
    • Enviando as atualizações para o repositório: 
      $ git push
  • Como verificar histórico de atualizações: 
    $ git reflog
    Serão exibidas todas as versões que foram "push-adas".
  • Como navegar entre as versões do seu código:
    • Quando foi rodado o comando "git reflog", o resultado foi esse:

      5d729ec ( HEAD -> master, origin/master ) HEAD@{0}: commit: Atualização 2022.03.29-08:46
      bde8524 HEAD@{1}: commit: commit inicial
      37884a6 HEAD@{2}: commit (initial): commit inicial

      Vamos imaginar que foi cometido um erro e será necessário voltar a uma versão anterior do código. Pra fazer isso, identifique o ID desejado, destacado acima em amarelo.
      $ git reset--hard bde8524

      RESULTADO:

      bde8524 (HEAD -> master) HEAD@{0}: reset: moving to bde8524
      5d729ec (origin/master) HEAD@{1}: commit: commit inicial
      37884a6 (HEAD -> master) HEAD@{2}: commit (initial): commit initial

      Pra voltar pra versão mais atual, é só repetir os passos acima passando o ID da sua última versão.
      A versão mais atual é mostrada pela destacamento: ( origin/master ) .

      Esse procedimento irá alterar o código LOCALMENTE.
  • O que são Branches e como criar uma:

    São caminhos diferentes que você vai seguir durante o seu processo de Desenvolvimento.

    • Ver quais branches estão disponíveis no momento: 
      $ git branch
    • Para criar uma nova branch: (NOTA: "staging" é o nome escolhido para a nova branch) 
      $ git branch staging

      * master
        staging

      A branch marcada com o asterisco(*) é a branch corrente (a branch que você está).

    • Para mudar de branch: 
      $ git checkout staging

        master
      * staging

      Note que agora é a branch 'staging' que está selecionada.

    • Agora vamos fazer o processo pra subir a versão do projeto definida para a branch 'staging'. 
      $ git add .
      $ git commit -m "adicionando nova branch"
      $ git push
      Quando você cria uma branch nova e você quer subir essa branch pro repositório em nuvem, temos de rodar o comando (que o próprio git fornece):
      Note que que é dado o nome da nova branch (atual). 
      $ git push --set-upstream origin staging
  • Como unir o código - GIT MERGE
    • Unindo o código da branch atual (staging) com a branch principal (maaster):
      1. Entra na branch 'master' (a branch que vai receber a união dos códigos): 
        $ git checkout master
      2. Antes de executar o 'merge', rode o comando 'pull' para ter certeza que a sua versão 'master' está atualizada em relação ao que está no repositório, evitando "mergear" códigos obsoletos. 
        $ git pull
      3. No comando abaixo será refernciada a branch de onde os códigos serão puxados para se unir às informações da branch 'master' (que é minha branch atual: 
        $ git merge staging
        Será exibido um relatório dos arquivos que foram alterados:

        Updating 5d729ec..41589aa
        Fast-forward
        index.html | 135 +++++++++++++++++++++++++++++++++++++++++++++++++++++++++----
        1 file changed, 128 insertions(+), 7 deletions(-)

      4. Execute o push para gravar as alteraçoes: 
        $ git push
        Proto, agora é só dar uma conferida no repositório em nuvem para confirmar se as alterações foram feitas corretamente.
    • DICA
      Criar uma nova branch com base na branch principal(master) e já se posicionar nela:
      Nome da nova branch: 'sistema-de-login'
      Nome da branch base: 'master' 
      $ git checkout -b sistema-de-login master

       

  • Um 'merge' democrático - PULL REQUEST

    Esse processo será utilizado quando for necessário que alguém autorize as suas alterações, permitindo mergear o seu código à branch principal(master).

    1. Quando estiver em uma branch diferente da principal(master), faça o push normalmente.
    2. No GitHub (site), na pasta principal do repositório, será exibida uma mensagem dizendo que a determinada branch recebeu pushes.

      Considerando que quem fez o push foi a branch 'sistema-de-login', a mensagem exibida será:

      sistema-de-login had recent pushes less than a minute ago.

    3. Clique em "Compare & pull request"
    4. Em "Comaparing changes", selecione a branch 'master'(principal) que vai receber o "merge" da branch 'sistema-de-login'.
    5. Informe o título do commit e se for necessário, em "Write" descreva informações adicionais.
    6. Depois clique em "Create pull request"

    Pronto!!
    Já está tudo "ok" para quem tiver permissão, revisar o código e aprovar ou não o "merge".

    Os Usuários que que estiverem marcados como "Reviewers" poderão aprovar clicando em "Merge pull request".
    Se vc estiver trabalhando "solo", poderá aprovar sua própria "pull request", se estiver em equipe, somente o(s) Revisor(es) terá(ão) esse 'direito'.

    Quando o "pull request" for aceito, será exibida uma mensagem:

    Pull request successfully merged and closed

  • GitIgnore - Como excluir arquivos mdo controle de versão

    • Para criar o arquivo .gitignore, rode o comando: 

      $ touch .gitignore

      Dentro do arquivo .gitignore vamos definir quais pastas e/ou arquivos não queremos que sejam adicionados ao controle de versão.
      Nesse exemplo, vamos adicionar a pasta 'segredos' da branch 'sistema-de-login' ao .gitignore para exclui-la do controle de versão.

      Para isso, edite o arquivo .gitignore, informe: 

      segredos/
fotos.png
      Para o caso de arquivos específicos, deve-se informar o path completo. Para o exmeplo do 'fotos.png', não foi incluído nenhum path, pois o arquivo está no diretório 'root'.

      Protno!! Agora a pasta e todo o seu conteúdo não aparecerão na listagem para controle de versão.

      Para confirmar, rode o comando: 
      $ git status
      Ela não vai mais aparecer marcada para ser adicionada
